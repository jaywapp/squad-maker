# 수익화 전략 제안서 리뷰 피드백

- 작성일: 2026-07-18
- 대상 문서: [2026-07-18-monetization-strategy.md](./2026-07-18-monetization-strategy.md)
- 리뷰 방식: 문서 논리 검토 + 현재 코드(`index.html`)와의 기능 대조
- 결론: **전략 방향과 문서 품질은 좋음. 핵심 수정 3건 반영 후 실행 문서로 사용 가능**

## 1. 총평

전략적 뼈대 — 무료 제작 도구로 획득, 공유로 활성화, 반복 사용·협업에 과금 — 는 유지할 가치가 충분하다.
다만 실행 단계에서 충돌할 내부 모순 1건과, 한국 시장에서 유료 서비스를 운영할 때의
현실 요건 누락이 있어 아래 세 가지 수정이 필요하다.

| # | 수정 항목 | 심각도 |
|---|---|---|
| 1 | §6 과금표가 기존 무료 기능(GIF·무제한 패턴)을 회수 — §12 위험 대응과 자기모순 | 높음 |
| 2 | §8 백엔드 요건에 한국 사업·법무 요건(사업자등록, PG, 개인정보보호법) 누락 | 높음 |
| 3 | §5 가격 가설만 있고 비용·손익분기 가설 부재 | 중간 |

## 2. 잘된 점

- **과금 대상 프레임이 정확하다.** "그리는 행위가 아니라 저장·재사용·전달·확인 과정에 과금"(§1)은
  제품 특성과 맞고, 경쟁 비교(§3)에서 종합 팀 관리 앱이 아니라는 자기 인식이 있어
  가격 포지셔닝 논리가 선다.
- **바이럴 루프 보호가 정합적이다.** 링크 열람 무료·비회원 유지(§4.1)는 배포된 뷰어 모드와 맞고,
  URL 인코딩 공유의 하위 호환 유지(§8)도 명시돼 있다.
- **절제된 실행 순서.** 0단계 fake-door 수요 측정 → 결제, Club은 수동 상담으로 수요 검증 후 개발,
  영구 이용권 배제 — 솔로 사이드 프로젝트에 맞는 접근이다.
- **가격 산식 검증 완료.** 연 59,000원은 월 결제(82,800원/년) 대비 28.7% 할인으로 "약 29%" 표기가 맞다.

## 3. 핵심 문제 1 — 자기모순: 기존 무료 기능을 회수하는 과금표 (§6 ↔ §12)

§12 첫 번째 위험의 대응은 "**기존 기능은 유지하고** 새 클라우드·협업 가치에 과금"인데,
§6 과금표가 그 반대를 하고 있다. 현재 코드 기준 전원 무료인 기능과 대조:

| 기능 | 현재 (전원 무료) | 문서의 비회원 등급 |
|---|---|---|
| GIF 내보내기 | `exportGif()` — 단일/전체 패턴 모두 지원 | 없음 |
| 다단계 패턴 | `addPattern()` — 패턴·단계 무제한 | 1패턴·3단계 |
| 무료 회원 GIF | (해당 없음 — 현재 전원 무료) | "체험 1회 검토" |

PR #26·#27로 이미 배포된 기능을 회수하는 것은 문서가 스스로 경고한
"무료 기능 축소로 기존 사용자 이탈" 위험 그 자체다.

### 권장 수정

둘 중 하나를 명시적으로 선택한다.

1. **(권장) 기존 기능 그랜드파더링**: GIF·무제한 패턴은 로컬 사용에 한해 계속 무료.
   과금은 신규 가치에만 — 클라우드 저장, 고해상도/영상 내보내기, 선수별 브리핑, 열람 확인, 버전 이력.
2. **의도적 회수**라면 §12 위험 표에 회수 사실과 이탈 완화책(기존 사용자 유예 기간 등)을 정직하게 기재.

## 4. 핵심 문제 2 — 한국 결제·법무 현실 누락 (§8)

§8 백엔드 목록은 기술 요소만 다룬다. 한국에서 유료 구독을 운영하는 순간 발생하는 요건:

- **사업자등록 + 통신판매업 신고** — 유료 구독 판매의 전제 조건.
- **PG 연동 현실** — 국내 카드 정기결제는 토스페이먼츠/포트원 등 국내 PG가 필요.
  Stripe는 국내 원화 정기결제가 사실상 어렵다. PG 수수료(~3%)와 고정비를 비용 모델에 반영해야 한다.
- **개인정보보호법** — 선수 이름을 서버에 저장하는 순간 개인정보 처리자가 되어
  개인정보처리방침 공개가 의무화된다. 아마추어 팀에는 **미성년 선수**가 흔해
  법정대리인 동의 이슈까지 검토가 필요하다. §8-7 "최소 정보 수집"만으로는 부족하다.

### 권장 수정

§8에 "사업·법무 요건" 항목을 추가하고, 1단계(무료 계정) 착수 전 완료 조건으로 명시한다.

## 5. 핵심 문제 3 — 손익분기 계산 부재 (§5)

가격 가설은 있으나 비용 가설이 없다. 현재는 GitHub Pages로 운영비 0원이지만,
1단계부터 다음 고정비가 발생한다.

- 호스팅 + DB + 인증 (예: Supabase/Firebase 유료 티어 진입 시 월 3~7만원 수준)
- PG 고정비 + 결제 수수료
- 도메인, 이메일 발송 등 부대 비용

"결제 의향 2~5%" 가설만 있고 분모(필요 트래픽)가 없어, 0단계 측정 결과를 보고
진행/중단을 판단할 기준이 서지 않는다.

### 권장 수정

§5에 손익분기 가설 한 단락 추가: 월 고정비 추정치 → 월 6,900원 기준 필요 구독자 수 →
전환율 2~5% 가설 역산 시 필요한 주간 활성 회원 수. 이 수치가 0단계의 go/no-go 기준이 된다.

## 6. 소소한 지적

- **§9 일정**: 단계당 3~5주는 풀타임 기준으로도 빠듯하다. 사이드 프로젝트면 2~3배로 보는 것이
  현실적이며, 특히 2단계 "구독 결제 + 해지·복구"는 구현보다 운영(환불 CS, 결제 실패 재시도)이 무겁다.
- **§10 분석 도구 미지정**: 이벤트 목록은 좋으나 0단계에서 쓸 도구가 없다. 정적 사이트라
  클라이언트 사이드만 가능 — GA4는 쿠키 동의 이슈가 있으니 Plausible/Umami 같은
  쿠키리스 도구를 후보로 명시할 것.
- **§13 북극성 지표**: "전술 준비~전원 확인 시간 절반"은 열람 확인 기능(3단계) 출시 전에는
  측정 불가. 초기에는 대리 지표(공유 링크 생성→열람 전환율)로 운영한다는 각주 필요.
- **문제없음 확인**: §4.1 비회원 "로컬 저장 최근 1개"는 현재 단일 문서 자동저장 모델과 동일해
  기능 회수가 아니다. `.sq` 파일 저장/불러오기도 실제 구현 확인(`saveSquadFile`/`loadSquadFile`).

2. 개정 후 0단계(수요 측정) 착수 여부 결정
