## 1. 종합 판정

작업 로그에 기재된 커밋과 산출물은 실제 브랜치 상태와 일치한다. Guest 회귀 계약, 분석 이벤트 사전, 설명형 수요 측정 UI, 인터뷰 스크립트가 계획서 Task 0.1~0.3에 맞춰 구현됐고 인증·클라우드 DB·결제는 추가되지 않았다.

다만 현재 분석 구현은 `websiteId`를 입력해 활성화할 경우 다음 문제가 발생할 수 있다.

1. Umami가 자동으로 붙이는 URL 정보에 `#s=` 공유 스냅샷이 포함될 가능성
2. Umami 스크립트가 준비되기 전에 발생한 이벤트의 유실 가능성

따라서 현 상태는 **“측정 UI와 이벤트 계약 구현 완료”**로 볼 수 있지만 **“실제 측정 시작 준비 완료”**로 보기는 어렵다. P1 수정과 활성화 상태 통합 테스트가 끝난 뒤 Umami 웹사이트 ID를 입력해야 한다.

## 3. 잘된 점

### 3.1 기존 무료 기능을 테스트 계약으로 먼저 고정

수익화 UI를 추가하기 전에 Guest 회귀 테스트를 만든 순서가 적절하다. 다음 기존 계약이 명시적인 배포 차단 대상으로 바뀌었다.

- LocalStorage `v:1` 복원과 자동 저장
- PNG 및 단일/전체 GIF 내보내기
- 패턴 추가
- `#s=` 공유 뷰어
- 공유 링크 열람자의 LocalStorage 보존
- `.sq` 저장·불러오기

`.sq` 테스트를 기본 스쿼드 초기화 동작에 맞춰 “상태 변경 후 원본 파일 복원” 방식으로 보정한 것도 실제 사용자 계약에 더 가깝다.

### 3.2 분석 이벤트의 명시적 사전

`ANALYTICS_EVENTS`에 등록된 이벤트와 속성만 `track()`이 복사하도록 해 자유 입력 텍스트가 이벤트 데이터에 섞이는 위험을 줄였다. 이벤트 값 40자 제한, 미등록 이벤트 폐기, `websiteId`가 없을 때 무전송으로 동작하는 구조도 합리적이다.

`_appReady`를 사용해 초기화·복원 과정의 자동 저장을 실제 편집으로 오인하지 않게 한 점도 퍼널 품질에 중요하다.

### 3.3 정직한 수요 측정 UI

- 기존 무료 버튼을 변경하거나 가로막지 않는다.
- 별도 `출시 준비 중` 영역에서만 노출한다.
- 출시 전 기능임을 명시한다.
- 현재 무료 기능이 유지된다고 설명한다.
- 예상 월 가격과 무료/Pro 범위를 표시한다.
- 공유받은 선수의 뷰어에는 노출하지 않는다.

Fake-door 실험에서 기능이 이미 존재하는 것처럼 오인시키지 않는 구성이 적절하다.

### 3.4 인터뷰 판정 기준

“좋다”는 반응과 실제 결제 의향을 구분하고 `누가, 어떤 예산으로, 언제 결제할지`가 나와야 Gate A 증거로 인정하도록 한 기준이 실용적이다.

## 4. 필수 수정 사항

### P1 — 분석 URL을 통한 공유 스냅샷 전송 가능성

#### 관찰

`index.html`의 동적 Umami 스크립트는 현재 `data-website-id`만 설정한다.

Umami는 기본 설정에서 페이지뷰를 자동 추적하고, 커스텀 이벤트에도 `url` 등 기본 속성을 추가한다. 또한 해시 수집은 별도로 끄지 않으면 활성 상태다. 따라서 앱의 이벤트 속성 화이트리스트와 무관하게 `#s=<encoded-state>`가 Umami 요청의 URL에 포함될 수 있다.

공식 근거:

- [Umami Tracker configuration](https://docs.umami.is/docs/tracker-configuration) — 자동 추적 기본 활성, `data-auto-track`, `data-exclude-hash` 설정
- [Umami Tracker functions](https://v2.umami.is/docs/tracker-functions) — 이벤트에 URL을 포함한 기본 속성이 추가됨

#### 영향

- 공유 전술 스냅샷이 국외 분석 서비스로 전송될 가능성
- 이벤트 사전의 “전체 공유 URL과 토큰은 전송하지 않는다” 계약 위반
- 작업 로그의 “화이트리스트가 개인정보·콘텐츠 미전송을 구조적으로 강제한다”는 설명과 실제 동작 불일치

#### 권장 수정

1. 분석 스크립트에 최소 `data-exclude-hash="true"`를 설정한다.
2. 수요 검증에 자동 페이지뷰가 필수가 아니라면 `data-auto-track="false"`도 적용한다.
3. `data-before-send` 또는 동등한 전송 직전 가드로 payload의 URL에서 hash와 query를 제거한다.
4. `#s=` 상태에서 분석을 활성화한 통합 테스트를 추가해 `/api/send` 요청 본문에 스냅샷·해시가 없음을 검증한다.

#### 완료 기준

- 일반 편집 화면과 `#s=` 뷰어의 실제 분석 요청에서 URL은 허용된 pathname만 포함한다.
- `#s=`, 팀명, 선수명, 지침, 스냅샷 일부를 요청 본문에서 검색했을 때 0건이다.
- 테스트가 어댑터의 `safe` 객체만 보지 않고 Umami로 나가는 최종 네트워크 payload를 검증한다.

### P1 — 분석 스크립트 로드 전 이벤트 유실

#### 관찰

`track()`은 `once` 요청을 받으면 먼저 `_trackedOnce`에 이름을 추가한 뒤 `window.umami?.track()`을 호출한다.

외부 스크립트가 아직 로드되지 않았다면 optional chaining으로 호출이 생략되지만 해당 이벤트는 이미 전송 완료로 표시된다. `share_link_opened`처럼 앱 초기화 직후 발생하는 이벤트는 이 경합에 특히 취약하다. 일반 이벤트도 준비 전 호출이면 그대로 사라진다.

#### 영향

- 공유 링크 열람 수가 실제보다 낮게 집계됨
- Gate A의 공유→열람 및 관심 퍼널 왜곡
- 네트워크 속도·광고 차단 상태에 따라 측정값이 달라짐

#### 권장 수정

1. 스크립트 준비 전 이벤트를 메모리 큐에 보관한다.
2. `script.onload`에서 순서대로 큐를 전송한다.
3. `once`는 큐 삽입 또는 성공적인 전달 상태를 기준으로 중복을 막되, 준비되지 않았다는 이유로 조용히 폐기하지 않는다.
4. 스크립트 로드 실패 시 앱에는 영향이 없어야 하며, 세션 중 재시도 여부를 명확히 정한다.

#### 완료 기준

- Umami 객체 생성을 지연시킨 테스트에서 초기 `share_link_opened`가 로드 후 정확히 1회 전달된다.
- 로드 전 같은 once 이벤트가 여러 번 발생해도 큐에는 1건만 남는다.
- 일반 이벤트는 발생 순서대로 전달된다.
- 스크립트 로드 실패 시 페이지 오류와 기능 장애가 없다.

## 5. 추가 개선 사항

### P2 — 배포 차단 테스트의 외부 CDN 의존성

#### 관찰

PNG·GIF 테스트는 다음 외부 리소스가 실시간으로 로드돼야 통과한다.

- `html2canvas.hertzen.com`
- `cdnjs.cloudflare.com`의 gif.js와 worker

검토 환경에서 `npm test`를 재실행했을 때 분석 테스트 9건까지 통과한 뒤 PNG와 GIF가 각각 30초·90초 타임아웃됐다. 이는 현재 환경의 외부 CDN 제한 영향으로 보이며, 이전의 `39 passed` 결과를 직접 반증하지는 않는다. 그러나 동일 커밋이 네트워크 조건에 따라 통과하거나 실패한다는 점은 배포 차단 테스트의 재현성을 낮춘다.

별도로 분석·관심 UI 20개 케이스는 모두 통과 출력을 냈지만 Playwright 웹 서버 종료가 지연돼 명령 자체는 타임아웃됐다.

#### 권장 수정

- 필수 Guest 계약 테스트는 고정 버전의 로컬 테스트 자산 또는 네트워크 route fixture로 실행한다.
- 실제 CDN 연결은 별도의 선택적 smoke 테스트로 분리한다.
- CI 기본 명령은 외부 네트워크 없이 결정적으로 끝나야 한다.
- Windows에서 Playwright `webServer` 종료가 지연되는 원인을 확인하고 `npx http-server` 대신 직접 바이너리 호출 또는 별도 서버 수명주기를 검토한다.

#### 완료 기준

- 네트워크를 차단한 상태에서 필수 테스트가 정해진 시간 안에 종료된다.
- CDN smoke 테스트의 실패는 앱 회귀와 공급자 장애를 구분해 보고한다.
- 테스트 종료 후 포트 4317과 Node 자식 프로세스가 남지 않는다.

### P2 — ADR의 동의 요건 단정

#### 관찰

ADR 0001은 쿠키와 핑거프린팅을 사용하지 않는다는 이유로 “별도 동의 배너 없이 사용할 수 있다”고 확정적으로 기술한다.

쿠키리스 여부는 중요한 기술 조건이지만, 실제 요청에 포함되는 URL·IP 처리·접속 메타데이터와 미국으로의 데이터 이전까지 검토하지 않은 상태에서 법적 결론을 확정하기에는 근거가 부족하다.

#### 권장 수정

- `동의 배너 불필요`를 `법무·설정 검토 후 확정`으로 변경한다.
- 수집 항목, 처리 목적, 보유기간, 처리 국가, 위탁/국외 이전 여부를 `privacy-data-map.md`에 기록한다.
- 0단계 활성화 전에 실제 네트워크 payload를 캡처해 문서의 수집 항목과 대조한다.

## 7. Gate A 상태

현재 Gate A는 **미통과가 아니라 아직 판정 전**이다.

| Gate A 조건 | 현재 상태 |
|---|---|
| 측정 오류율 5% 미만 | 측정 미활성, 산정 불가 |
| 4주 또는 완성 세션 200건 | 0건 |
| 활성 제작자 10% 이상 유료 후보 확인 | 측정 미활성 |
| 인터뷰 5~8건 중 3명 구체적 결제 의향 | 미실시 |
| Pro 25명 도달 가능한 유입 경로 | 미검증 |

P1 수정 → Umami 활성화 → 4주 측정·인터뷰 → Gate A 회의록 순서를 지켜야 한다. Gate A 판정 전에는 계획대로 인증·클라우드 DB·결제를 시작하지 않는다.

## 10. 최종 결론

0단계 구현의 방향, 작업 순서와 산출물 품질은 좋다. 특히 기존 무료 기능을 먼저 회귀 계약으로 고정하고, 기능을 약속하지 않는 설명형 UI로 수요를 측정한 결정은 계획과 일치한다.

그러나 분석 시스템의 개인정보 경계는 앱의 `track()` 함수만으로 끝나지 않는다. 외부 분석 스크립트가 자동으로 추가하는 URL 메타데이터와 로드 시점까지 포함해야 실제 전송 계약이 완성된다.

따라서 현재 판정은 다음과 같다.

> **0단계 코드 구현은 조건부 완료. P1 2건과 활성화 상태 통합 테스트를 해결한 뒤 측정을 시작한다.**
