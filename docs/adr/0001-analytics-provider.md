# ADR 0001 — 분석 도구 선정

- 상태: 승인 (2026-07-20)
- 관련 계획: [수익화 적용 계획서](../2026-07-20-monetization-implementation-plan.md) Task 0.2

## 배경

0단계의 목표는 백엔드를 만들기 전에 실제 사용과 결제 관심을 측정하는 것이다.
비교 기준은 계획서에 정의된 네 가지다: **비용, 운영 부담, 데이터 위치, 동의 요건**.

전제 조건:

- 앱은 GitHub Pages 정적 배포 — 서버 코드 없이 스크립트 삽입만 가능해야 한다.
- 0단계는 수익이 0이므로 고정비를 최소화한다 (운영비 상한 가설: 월 100,000원).
- 선수·팀 이름, 지침, 공유 스냅샷 등 개인정보·콘텐츠는 절대 전송하지 않는다.
- 분석 도구 차단·장애가 앱 기능을 방해하면 안 된다.

## 후보 비교

| 기준 | Plausible Cloud | Umami Cloud | Umami 자체 호스팅 |
|---|---|---|---|
| 비용 | 월 €9~ (10k 페이지뷰) | **무료 티어** (Hobby: 월 100k 이벤트, 웹사이트 3개) | 서버·DB 비용 + 관리 시간 |
| 운영 부담 | 없음 | 없음 | DB 백업·업데이트·모니터링 직접 부담 |
| 데이터 위치 | EU (독일) | 미국 (Cloud) | 선택 가능 |
| 동의 요건 | 쿠키·핑거프린팅 없음 (배너 요건은 검토 필요) | 쿠키 없음 (배너 요건은 검토 필요) | 동일 |
| 커스텀 이벤트 | 지원 (플랜 한도 내) | 지원 (이벤트 단위 과금 기준) | 지원 |
| 교체 용이성 | 스크립트 1개 | 스크립트 1개 | 스크립트 1개 |

두 클라우드 후보 모두 쿠키리스·비개인화 집계 방식이라 동의 배너 없이 운영할 수 있는
조건에 가깝다. 다만 **이는 아직 확정된 법적 결론이 아니다.** 쿠키 사용 여부만으로는
판단할 수 없고, 실제 요청에 포함되는 URL·IP 처리·접속 메타데이터와 미국으로의
데이터 이전까지 함께 검토해야 한다.

- 현재 상태: **법무·설정 검토 후 확정** (0단계 활성화 전 완료할 것)
- 검토 산출물: `docs/privacy-data-map.md`(Task 1.1)에 수집 항목·처리 목적·보유기간·
  처리 국가·국외 이전 여부를 기록하고, 실제 네트워크 payload를 캡처해 문서와 대조한다.
- payload 실측 근거는 `tests/e2e/analytics-payload.spec.js`가 상시 검증한다.

## 결정

**Umami Cloud (Hobby 무료 티어)** 를 사용한다.

- 0단계는 지출 없이 측정을 시작하는 것이 최우선이다. 예상 트래픽(월 수천 세션 이하)은
  무료 한도(100k 이벤트) 안에 충분히 들어온다.
- 데이터 위치(미국)는 개인정보를 보내지 않는 설계이므로 수용 가능하다.
  1단계에서 계정·개인정보를 다루기 시작하면 `docs/privacy-data-map.md`(Task 1.1)에서 재평가한다.
- 도구 종속을 피하기 위해 앱 코드는 `track(name, properties)` 어댑터 하나만 호출한다.
  Plausible로 교체 시 어댑터 내부와 스크립트 태그만 바꾸면 된다.

## 구현 규칙

- 이벤트·속성은 [analytics-event-dictionary.md](../analytics-event-dictionary.md)에 정의된
  화이트리스트만 전송한다. 사전에 없는 이벤트·속성은 어댑터가 폐기한다.
- **어댑터 화이트리스트만으로는 부족하다.** Umami는 모든 payload에 `url`·`referrer`를
  자동으로 붙이고 해시는 기본 포함이므로, `#s=` 공유 스냅샷이 그대로 전송된다.
  따라서 다음 3중 방어를 반드시 함께 적용한다.
  1. `data-exclude-hash="true"`, `data-exclude-search="true"`
  2. `data-before-send` 가드로 `url`을 pathname까지, `referrer`를 origin+pathname까지 축약
  3. 최종 네트워크 payload를 검증하는 통합 테스트(`tests/e2e/analytics-payload.spec.js`)
- 자동 페이지뷰 추적(`data-auto-track`)은 **끄지 않는다.** Gate A의 "활성 제작자의 10% 이상"
  판정에 방문·세션이라는 분모가 필요하다. 단일 페이지 앱이라 해시·쿼리를 제거하면
  경로에는 콘텐츠가 남지 않으므로, 위 방어와 함께 쓰면 안전하다.
- `websiteId`가 비어 있으면 스크립트를 로드하지 않고 아무것도 전송하지 않는다
  (기능 플래그 `analytics` 역할). 배포 시 `window.SQUAD_MAKER_ANALYTICS_ID`로 주입하거나
  `ANALYTICS_CFG.websiteId` 기본값에 직접 입력해 활성화한다.
- 스크립트는 `defer`로 로드되므로 초기화 시점에는 `window.umami`가 없다.
  준비 전 이벤트는 큐(상한 50건)에 보관했다가 `load`에서 발생 순서대로 전달한다.
  큐가 없으면 `share_link_opened`처럼 초기화 직후 발생하는 이벤트가 항상 유실된다.
- localhost에서는 콘솔(`[analytics]` 접두사)로도 출력해 개발 중 이벤트 발생을 검증한다.
- 스크립트 로드 실패·차단 시 앱 동작에 영향이 없어야 한다. 큐를 비우고 재시도하지 않는다.

## 결과

- 무료로 4주 측정을 시작할 수 있고, Gate A 판단에 필요한 퍼널 데이터를 확보한다.
- 트래픽이 무료 한도를 넘으면 그 자체가 수요 신호이므로 유료 전환 또는
  Plausible 재평가를 그 시점에 결정한다.
